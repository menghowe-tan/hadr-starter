01

The problem, in one breath

On day zero of a disaster, the information is fragmented, noisy, and constantly revised. Three feeds each tell a partial story at a different latency: GDACS and USGS give you raw physics in minutes, modeled impact in hours; ReliefWeb gives you human confirmation in days. None of them is a morning briefing.

To answer “what happened overnight, how bad is it, what changed since yesterday?” a person today has to read all three, mentally dedupe them, and remember what was already known. This system is an unattended agent that does exactly that reading — on a schedule, unprompted.

02

The one rule that shapes everything

Deterministic before generative

A deterministic script decides whether anything changed. A model runs only if it did. The model never decides whether to wake up.

This is the intuition behind the whole repo. Fetching, merging, dedup and change detection are things that must give the same answer twice — so they live in plain Python (scripts/) and never call a model. The model's job is narrower and more expensive: assessment and writing. Gating it behind a cheap, repeatable check keeps the system predictable, auditable, and cheap to run every few minutes.

04

The three feeds

Each has a different character — fast and raw, or slow and human-verified — and the system trusts each for what it's good at.

GDACSminutes · multi-hazard

EU/UN alerts across six hazard types (quake, cyclone, flood, volcano, drought, wildfire). Colour-coded Green Orange Red alert levels. Its severity is a model forecast, not an observation.

USGSevery minute · earthquakes

Real-time seismic feed, regenerated every minute. Fast and raw — events get revised and occasionally deleted. Identity is the union of its ids, never a single id, because the network prefix can flip.

ReliefWebdays · human-verified

UN OCHA's curated record. A disaster appears once people decide it matters — slow by design. Days-latency, so its absence never blocks a morning report.

Intuition · echo, not corroboration

GDACS's earthquake lane is itself USGS-derived (source: NEIC). When the two “agree,” that's an echo of the same source — the sitrep must never present it as two feeds confirming each other.

05

The severity gate

The gate is deterministic. The model may add a notable event (with an explicit “editorial” label) but may never remove one. What always clears the bar:

  • Any hazard at Orange or Red alert level (event- or episode-level).
  • Earthquakes with PAGER ≥ yellow, significance ≥ 600, or magnitude ≥ 5.5 shallower than 70 km.
  • Temporary and below-threshold events are filtered out — deterministically, before any model sees them.

Intuition · estimated vs reported

GDACS scores and USGS PAGER are forecasts, not counts. Every impact figure is labelled Estimated (a model said so) or Reported (a human confirmed it). A bare number is never allowed.

06

Identity & merge — one event, many records

Two records become one canonical event by the first tier that matches:

Tier 1 · confirmed

GLIDE numbers equal. The strongest tie — but reliable only in ReliefWeb, and it arrives days late.

Tier 2 · high

Same hazard, origin times within 30 min, epicentres within 100 km.

Tier 3 · possible

Same hazard + country within 24 h. Surfaced as a maybe — shown, not merged silently.

Who decides

A script decides every merge. The model only explains it. Deterministic before generative, again.

07

Two cadences

The same pipeline runs on two clocks — and only one of them ever wakes the model.

EVERY ~5–15 MIN · NO MODEL

Monitor loop

Fetch → merge → diff, then redeploy the live dashboard whenever the store changes. Every marker and freshness stamp reflects the latest fetch. Purely deterministic.

08:30 SGT · MODEL WAKES

Daily sitrep

Once a day, gates on what changed since the previous sitrep and — only then — wakes the model to write the plain-language morning briefing. The single place a model writes.

Time is normalised to UTC at fetch time; Singapore time appears only in the rendered view. “Since last report” always means a diff against our own stored state, never against a re-fetched rolling window — that's how boundaries stop double-counting.

08

Quiet, degraded, down — trusting silence

A quiet report has to be a statement, not a failure. So the daily run has three honest end states:

ALL QUIET

All feeds fetched, gate says nothing changed → publish an explicit “all quiet” sitrep with a feed-health banner. Silence you can trust.

DEGRADED

One feed stale, others fresh → publish anyway, with a prominent per-feed warning saying exactly what's missing.

ABORT

GDACS and USGS both unreachable → no sitrep; fail loudly (exit code 3). A blind morning is worse than no report. ReliefWeb alone can't see today.

09

Harness vs. agent, data vs. views

harness/

A reusable, project-agnostic agent harness — chat loop, standing orders, tools, agent loop. Zero repo imports; copy it into any project.

agent/

Where the harness is wired to HADR — the disaster tools and the standing objective in goal.md. Nothing HADR-specific leaks into the harness.

Intuition · committed JSON is the database

Canonical merged events live as one JSON file per event under data/, and git history is the audit trail. The sitrep HTML and the map are generated views — never committed; they deploy to an external host.
